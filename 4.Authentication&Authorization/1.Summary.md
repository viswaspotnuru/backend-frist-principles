In this session, Srini breaks down the core concepts of **Authentication** (Who are you?) and **Authorization** (What can you do?) by exploring their historical evolution and modern technical implementations.



---

### **1. Core Definitions**
* **Authentication (AuthN):** The mechanism used to assign an identity to a subject. It answers: *"Who are you?"*
* **Authorization (AuthZ):** The process of determining a user's permissions. It answers: *"What are you allowed to do?"*.

### **2. Essential Components**
* **Sessions:** Since HTTP is stateless, the server creates a **Session ID** upon login and stores user data in a fast database like **Redis**.
* **JWT (JSON Web Tokens):** A stateless way to verify identity. It contains a Header, a Payload (user info), and a Signature (to prevent tampering).
* **Cookies:** A browser mechanism used to store the Session ID or JWT. The `HttpOnly` flag is critical to prevent JavaScript from stealing the token.

### **3. Implementation Examples**

#### **Golang (Gin Framework)**
In Gin, you typically use **Middlewares** to check for a token in the `Authorization` header.

```go
// Middleware to check JWT
func AuthMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        token := c.GetHeader("Authorization") // Reading the header [1:22:15]
        if token == "" {
            c.AbortWithStatusJSON(401, gin.H{"error": "Authentication failed"})
            return
        }
        // Validate token logic here...
        c.Next()
    }
}
```

#### **Node.js (Express)**
Standard implementation involves checking for a Bearer token.

```javascript
const jwt = require('jsonwebtoken');

const verifyToken = (req, res, next) => {
  const authHeader = req.headers['authorization']; // [1:23:40]
  const token = authHeader && authHeader.split(' ')[1];

  if (!token) return res.status(401).send('Access Denied');

  jwt.verify(token, process.env.TOKEN_SECRET, (err, user) => {
    if (err) return res.status(403).send('Forbidden'); // AuthZ Error [1:21:50]
    req.user = user;
    next();
  });
};
```

#### **Python (Django)**
Django uses `IsAuthenticated` permission classes and settings to manage which users can access specific views.

```python
from rest_framework.permissions import IsAuthenticated
from rest_framework.views import APIView

class UserProfileView(APIView):
    # This automatically checks if the user is logged in (AuthN)
    permission_classes = [IsAuthenticated] 

    def get(self, request):
        # Role-Based Access Control (AuthZ) [1:19:18]
        if not request.user.is_staff:
            return Response({"error": "Forbidden"}, status=403)
        return Response({"data": "Admin content"})
```

### **4. Critical Security Strategies**
* **Generic Error Messages:** Srini warns against saying "User not found" or "Incorrect password." Always use a generic message like **"Authentication failed"** to prevent attackers from guessing valid usernames.
* **Timing Attacks:** Attackers measure how long the server takes to respond to deduce valid usernames. 
    * *Solution:* Backend engineers should use **constant-time operations** or simulate a fake delay for failed requests so that every response takes the same amount of time.

### **5. Summary Table: Which to Choose?**

| Method | Best For | Pros | Cons |
| :--- | :--- | :--- | :--- |
| **Sessions** | Traditional Web Apps | Easy to revoke access | Hard to scale across multiple servers |
| **JWT** | Microservices / Mobile | Highly scalable and stateless | Hard to revoke before expiration |
| **OAuth 2.0** | Third-party integrations | Safe data delegation | High complexity to implement |
| **API Keys** | Machine-to-Machine | Fast and programmatic | Risk if key is exposed in code |
