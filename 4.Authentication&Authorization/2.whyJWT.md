The transition from **Stateful Sessions** to **JWT (JSON Web Tokens)** happened primarily to solve the **Scalability** and **Latency** challenges that arose as web applications grew from single servers to global, distributed systems.

In the video [Authentication and authorization for backend engineers](https://www.youtube.com/watch?v=A95rliroC8Q), Srini explains that this shift was a move from a "Stateful" model to a "Stateless" one.

### **1. The Problem with Sessions: Scalability**

In a traditional **Stateful** setup:

  * **Centralized Storage:** Every time a user logs in, the server creates a session and stores it in an in-memory database like **Redis**.
  * **The Bottleneck:** As you scale to millions of users across multiple regions (e.g., AWS US-East and AWS EU-West), every single request requires the server to "look up" the session in that central database to verify who the user is.
  * **Synchronization Issues:** Synchronizing these session databases across global data centers introduces significant **latency** and operational complexity.

### **2. The Solution: Statelessness with JWT**

JWTs solved this by offloading the "memory" from the server to the client.

  * **Self-Contained Data:** A JWT contains all the necessary user information (ID, roles, etc.) inside its **Payload**.
  * **Cryptographic Trust:** Instead of checking a database, the server uses a **Secret Key** to verify the token's **Signature**. If the signature is valid, the server *trusts* the data inside without needing to look anything up.
  * **No Database Lookup:** This allows any server in any part of the world to verify a user instantly, making it highly efficient for **Microservices** and globally distributed apps.

### **3. Trade-offs to Consider**

While JWTs improved scalability, they introduced new challenges that Srini highlights:

  * **Revocation is Hard:** Because the server doesn't check a database, you cannot easily "log out" a user or revoke a token before it expires. With sessions, you just delete the entry from Redis. With JWTs, the token remains valid until its expiration time (`exp`) is reached.
  * **Size:** JWTs are sent with every request header. If you put too much data in the payload, it can increase the overhead of every HTTP request compared to a tiny Session ID.

### **Summary of the Transition**

| Feature | Stateful Sessions (The Old Way) | Stateless JWT (The New Way) |
| :--- | :--- | :--- |
| **Storage** | Server-side (Redis/DB) | Client-side (Encoded in token) |
| **Scalability** | Difficult (DB is a bottleneck) | High (No DB lookup needed) |
| **Revocation** | Instant (Delete session) | Difficult (Must wait for expiry) |
| **Best For** | Monolithic Web Apps | Microservices & Mobile Apps |
