While JWT (JSON Web Tokens) solved the massive scalability issues of stateful sessions, they introduced a specific set of "backend headaches" that every engineer must handle. 

Srini discusses these trade-offs in the **[Authentication and Authorization](https://www.youtube.com/watch?v=A95rliroC8Q)** video. Here are the primary issues with JWT:

---

### **1. The Revocation Problem (The "Kill Switch" Issue)**
Because JWTs are **stateless**, the server doesn't check a database to see if the token is still "valid"—it only checks if the cryptographic signature is correct.
* **The Issue:** If a user’s phone is stolen or an admin wants to ban a user immediately, a JWT cannot be easily "deleted." The token will remain perfectly valid until its expiration time (`exp`) is reached.
* **The Fix:** Engineers often use a **Deny-List** (storing blocked tokens in Redis) or move to a **Hybrid Model** where short-lived JWTs are paired with long-lived "Refresh Tokens" that can be revoked.

### **2. Stale Data**
The data inside a JWT (the **Payload**) is encoded when the token is created.
* **The Issue:** If a user’s permissions change (e.g., they are promoted from `user` to `admin`), their current JWT still says they are a `user`. They won't see the change until they log out and get a new token.
* **The Fix:** Keep JWT lifespans very short (e.g., 15 minutes) so the data is refreshed frequently.

### **3. Security: The "Standard" Misconception**
A common mistake beginners make is thinking JWTs are **encrypted**.
* **The Issue:** JWTs are only **Base64 encoded**, not encrypted. Anyone who intercepts a JWT can see the payload (User ID, Email, etc.) using tools like [JWT.io](https://jwt.io).
* **The Fix:** **Never** put sensitive data like passwords or social security numbers in a JWT payload. Always send JWTs over **HTTPS** to prevent interception.

### **4. Token Size & Overhead**
* **The Issue:** Unlike a Session ID (which is a tiny string), a JWT can become quite large if you store many "claims" (user roles, preferences, etc.). Since this token is sent in the header of **every single HTTP request**, it increases bandwidth usage.
* **The Fix:** Keep the payload lean. Only store essential identification data.

---

### **Comparison: Sessions vs. JWT Issues**

| Issue | Stateful Sessions | Stateless JWT |
| :--- | :--- | :--- |
| **Storage** | Heavy on Server (Redis/DB) | Heavy on Network (Header size) |
| **Revocation** | Easy (Delete from DB) | Hard (Wait for expiry) |
| **Complexity** | Centralized bottleneck | Distributed security risks |
| **Outdated Info** | Never (DB is always current) | Possible (Payload is a snapshot) |



### **Practical Implementation: Security Headers**
To mitigate JWT theft, Srini emphasizes setting these headers when storing tokens in cookies:
* **`HttpOnly`:** Prevents JavaScript from accessing the token (stops XSS attacks).
* **`Secure`:** Ensures the token is only sent over HTTPS.
* **`SameSite=Strict`:** Prevents the token from being sent in cross-site requests (stops CSRF attacks).
