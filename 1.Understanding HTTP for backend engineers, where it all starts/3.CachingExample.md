In the video, Srini breaks down caching into two stages: the **Initial Request** (where the server sets the rules) and the **Conditional Request** (where the client checks if the data is still valid).

-----

### **1. The Initial Request**

The client asks for a resource for the first time. The server responds with the data and provides "metadata" headers that the browser will save for future use.

| Side | Header | Example Value | Description |
| :--- | :--- | :--- | :--- |
| **Server** | `Cache-Control` | `max-age=10` | Tells the browser to keep this file for 10 seconds. |
| **Server** | `ETag` | `"3141"` | A unique "fingerprint" (hash) of the current data. |
| **Server** | `Last-Modified` | `Wed, 21 Oct 2024...` | The timestamp of when the file was last updated. |

-----

### **2. The Conditional Request (After 10 seconds)**

Once the `max-age` expires, the browser doesn't just delete the file. It sends a **Conditional GET** to the server, providing the saved metadata to see if the file has changed.

| Side | Header | Example Value | Description |
| :--- | :--- | :--- | :--- |
| **Client** | `If-None-Match` | `"3141"` | "I have version 3141. Is there anything newer?" |
| **Client** | `If-Modified-Since` | `Wed, 21 Oct 2024...` | "Has this changed since this specific time?" |

-----

### **3. The Server's Decision**

#### **Scenario A: No Changes Found**

If the server sees that the `ETag` "3141" is still current, it saves bandwidth by sending **no body**.

  * **Response Status:** `304 Not Modified`
  * **Result:** The browser continues using its old local copy.

#### **Scenario B: Data Has Been Updated**

If the data was edited on the server, the new `ETag` (e.g., `"2943"`) won't match the client's.

  * **Response Status:** `200 OK`
  * **Headers:** Server sends a **new `ETag`** and a **new `Last-Modified`** timestamp.
  * **Result:** The server sends the entire new file, and the browser replaces its old cache.
