A **Simple Request** in CORS occurs when a browser makes a cross-origin request using standard methods and headers, allowing the browser to skip the "pre-flight" (OPTIONS) check.


### **Example Scenario [21:44](http://www.youtube.com/watch?v=a3C1DMswClQ&t=1304)**
Imagine your frontend (Client) is running on `http://example.com` and your backend (Server) is at `http://api.anotherdomain.com`.

1.  **Client Sends Request:** The client initiates a `GET` request to the server.
2.  **Browser Adds Origin:** The browser automatically attaches an `Origin` header to the request to identify where the request is coming from:
    * `Origin: http://example.com`
3.  **Server Checks Policy:** The server receives the request and checks the `Origin` header against its CORS policy.
4.  **Server Responds:** If allowed, the server includes the `Access-Control-Allow-Origin` header in its response:
    * `Access-Control-Allow-Origin: http://example.com` (or `*` for all origins).
5.  **Browser Validation:** The browser receives the response and checks for that specific header.
    * **Success:** If the header matches the client's origin, the browser allows the JavaScript code to access the data [23:22](http://www.youtube.com/watch?v=a3C1DMswClQ&t=1402).
    * **Failure:** If the header is missing or does not match, the browser blocks the response and throws a **CORS error** in the console [24:00](http://www.youtube.com/watch?v=a3C1DMswClQ&t=1440).

### **Conditions for a Simple Request [21:14](http://www.youtube.com/watch?v=a3C1DMswClQ&t=1274)**
For a request to be "Simple," it must meet these criteria:
* **Methods:** Only `GET`, `POST`, or `HEAD`.
* **Headers:** Only standard headers (e.g., `Accept`, `Language`, `Content-Type`).
* **Content-Type:** Must be one of `application/x-www-form-urlencoded`, `multipart/form-data`, or `text/plain`.

If your request uses `application/json` or a `PUT` method, it will trigger a **Pre-flight Request** instead [24:14](http://www.youtube.com/watch?v=a3C1DMswClQ&t=1454).

**Preflight Request**
In a CORS (Cross-Origin Resource Sharing) flow, a **Preflight Request** is a safety check the browser performs before sending the actual request to a different domain. It ensures that the server understands and permits the specific "non-simple" actions the client is about to take.



### **Why is it needed?**
Standard requests (Simple Requests) like a basic `GET` are generally safe. However, requests that could modify server data (like `PUT` or `DELETE`) or those sending custom data (like `application/json`) require the browser to "ask for permission" first to prevent unauthorized actions.

### **The OPTIONS Method**
The preflight check is always performed using the **OPTIONS** method. This is a "general inquiry" that does not contain any actual data (no request body). It simply asks the server: *"What are your capabilities for this specific URL?"*

---

### **Preflight Example**

Imagine a client at `https://example.com` wants to update a user profile on an API at `api.anotherdomain.com` using a `PUT` method and an `Authorization` header.

#### **1. The Preflight Request (OPTIONS)**
The browser automatically fires this request first:

**Request:**
```http
OPTIONS /api/resource HTTP/1.1
Host: api.anotherdomain.com
Origin: https://example.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Authorization
```
* **Access-Control-Request-Method:** Tells the server the client wants to use `PUT`.
* **Access-Control-Request-Headers:** Tells the server the client wants to send an `Authorization` header.

#### **2. The Server's Response**
If the server allows this cross-origin interaction, it responds with a **204 No Content** status and specific headers:

**Response:**
```http
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400
```
* **Access-Control-Allow-Origin:** Confirms the client's domain is allowed.
* **Access-Control-Max-Age:** Tells the browser it can cache this "permission" for 24 hours (86400 seconds) so it doesn't have to keep sending preflight requests for every single action.

#### **3. The Original Request**
Once the browser validates the `OPTIONS` response, it finally sends the actual `PUT` request the developer originally intended.

---

### **When is a Preflight Triggered?**
A request qualifies for a preflight if **any** of these are true:
1.  The method is **NOT** `GET`, `POST`, or `HEAD` (e.g., it is `PUT`, `PATCH`, or `DELETE`).
2.  The request includes **Custom Headers** (like `Authorization` or `X-Custom-Header`).
3.  The `Content-Type` is anything other than the simple types (most commonly triggered when using **`application/json`**).
