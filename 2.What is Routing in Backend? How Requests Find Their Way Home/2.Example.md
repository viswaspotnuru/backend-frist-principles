Deciding when to use a specific type of route depends on the **semantic meaning** of the action you are performing and the **nature of the data** you are passing.

Based on the [What is Routing in Backend?](https://www.youtube.com/watch?v=SubuU1iOC2s) video, here is a guide on how to choose:

---

### **1. Use Static Routes for Collections or General Actions**
Use a [Static Route](https://www.youtube.com/watch?v=SubuU1iOC2s&t=280s) when the path refers to a stable resource or a general "bucket" of data that doesn't change based on a specific ID.
* **When to use:** Listing all items or creating a new one.
* **Example:** `GET /api/books` (Fetch all) or `POST /api/books` (Create new).

### **2. Use Dynamic Routes (Path Parameters) for Specific Resources**
Use [Dynamic Routes](https://www.youtube.com/watch?v=SubuU1iOC2s&t=323s) (using `:id` placeholders) when the URL needs to identify a **unique entity**.
* **When to use:** When the value is required to locate the resource. If the request doesn't make sense without that ID, it belongs in the path.
* **Example:** `GET /api/users/123`. The `123` is a [Path Parameter](https://www.youtube.com/watch?v=SubuU1iOC2s&t=528s) that semantically identifies exactly *which* user you want.

### **3. Use Query Parameters for Metadata and Filtering**
Use [Query Parameters](https://www.youtube.com/watch?v=SubuU1iOC2s&t=615s) (`?key=value`) when the information is **optional** or used to **modify the result set** rather than identify a single resource.
* **When to use:**
    * **Searching/Filtering:** `GET /api/books?author=rowling`.
    * **Sorting:** `GET /api/books?sort=asc`.
    * **Pagination:** `GET /api/books?page=2&limit=20`.
* **Rule of Thumb:** If the parameter can be omitted and the request still "works" (returns a default list), use a query parameter.

### **4. Use Nested Routes to Show Relationships**
Use [Nested Routing](https://www.youtube.com/watch?v=SubuU1iOC2s&t=909s) when one resource "belongs" to another.
* **When to use:** To provide a clear hierarchy and semantic expression of ownership.
* **Example:** `GET /api/users/123/posts/456`. This clearly communicates you are looking for "Post 456 belonging to User 123".

---

### **Decision Summary Table**

| Scenario | Recommendation | Why? |
| :--- | :--- | :--- |
| **Fetch one specific user** | **Path Parameter** (`/users/123`) | Identifies a unique resource semantically. |
| **Filter a list by category** | **Query Parameter** (`?cat=tech`) | Modifies the view of the list; optional. |
| **Get comments for a photo** | **Nested Route** (`/photos/1/comments`) | Shows the relationship (Ownership). |
| **Request page 5 of results**| **Query Parameter** (`?page=5`) | It's metadata about the response, not the resource itself. |
| **New API Version** | **Versioning** (`/v2/users`) | Prevents breaking changes for existing clients. |



By following these patterns, you ensure your API remains **human-readable** and follows industry-standard **REST principles**.
